Arrays

Nimble Basic Concepts:

Concept
Definition / Properties / Use
Group
  • Logical entity used to represent a collection of physical arrays.
  • Arrays in a group can be managed together via GUI/CLI/REST API. (single point of management)
  • Maximum of 4 Arrays in a group. Most customers have 1 array in a pool/group except in transient array replacement cases


  • in a Nimble Group there is exactly one FC (Fiber Channel) Target identified by the WWN (world-wide node name) and one iSCSI target identified by the IQN (iSCSI Qualified Name)
Pool
  • A collection of one or more arrays in which volumes reside. Volumes are carved out of pools.
  • Using pools, a group can be divided into pools containing one or more arrays.
  • An array can belong to exactly one pool.
  • Arrays can be grown by adding shelves (capacity growth)
  • Add arrays to a pool to grow capacity or performance.
  • Max Pool count is 4 (in which case you would have one array per pool). Most customers have 1 array in a pool/group except in transient array replacement cases
  • customers can have mixed array types/models in a single pool
Folder
  • used within a pool to organize volumes
Volume
  • logical representation of blocks in a pool in which to read/write data
  • things like snapshots & clones act on volume collections
  • "Volume Scoped Target" - separate target for each volume. Intitiator access is controlled by ACLs

Nimble Array



Physical Components

  • single box containing a collection of disks (HDDs and/or SSDs)
  • Purpose: to store and serve data blocks
  • Physical machine that runs Nimble Software
  • Has 2 controllers (active and standby)
Groups and Pools
  • collection of arrays (max of 4 per group but most customers have 1 array in a pool/group except in transient array replacement cases)
  • customer configured
  • add shelves to array for storage capacity
  • add more arrays for performance
Controller
  • CPU
  • Memory
  • Network ports
  • NVRAM
  • Disks (SSD, HDD) are shared between controllers
  • Usually controllers are mirror images for HA (High Availability)
    • Exception is transient conditions like upgrade
  • 2 controllers for no SPOF (Single Point of Failure)
  • Active/Passive configuration: only one controller has access to the backend at any point in time
  • Standard Linux Kernel
  • Customization for SW RAID & NVRAM drivers
  • Magic is in User-mode processes
    • for easy dev, debug, patch

ATM

Asynchronous Task Manager (management) used by NIMOS Data Services (and gmd)

  • manages multiple operations on an array
  • dedicated thread pool multiplexing
  • spawns child ATM ops
  • message passing framework
  • fixed size: 700 bytes
  • an ATM op combines state and state machine
    • no matter where you are in the stack we use ATM ops
    • Eg. ATM op to do SCSI READ with states: read from socket, dispatch to VolMgmt layer, Block waiting for VolMgr op, write to TCP socket
  • CPU scheduler for ATM ops (which CPU and when)
  • ATM tasks have IDs for QoS management (noisy neighbor; balance ops to manage for required behavior)

CASL (tm)

Cache Accelerated Sequential Layout (Nimble trade mark)

  • only writes to completely empty segments
  • CASL doesn't have files. It has volumes which are address spaces. Broken up into blocks with user configured blocksize
  • Purpose: to provide efficient performance on Hybrid systems (HDs) by converting random updates that come into NVRAM to sequential writes that are packed into segments
  • CASL Tasks (DBB Sync)
    1. writes come in and hang out in NVRAM
    2. then DBB Sync gathers data from NVRM and chunks it up into CASL blocks
    3. For each block it
      1. compresses it
      2. computes a dedupe fingerprint (eg. SHA256)
      3. looks it up in the Fingerprint Index (FI) to decide it it is a duplicate
      4. If it is a new block it assigns an SBN (sequential block number) [BU = SBN + domain)
      5. Add SBN to the index and choose an LFS location on disk
      6. when 10 MBs are gathered it flushes the segment to disk

GC

Garbage Collection/Sweeper: The Garbage Collection subcomponent of a Nimble array that generates completely-empty segments for writing into.

  • picks low-utilization segment
  • identifies and copies live data to new segment
  • frees the old segment
  • done on flash and HD

Asynchronous process to generate free segments

  • GC policies manage frequency

Nimble Ref: DsdDataflow

The garbage collector is what frees up segments. GC tracks the amount of live bytes in every utilized segment in a Nimble array, and is notified whenever a segment is written or a block is deleted. The GC monitor runs periodically and identifies a target for the the minimum number of free segments there should be on the array; this target is determined based on configuration variabls and the amount of user data on the system (very full systems have a lower target). When the number of free segments drops below this target, SL notifies GC, and GC then picks a collection of segments to reap.

Segments are selected for reaping based both on their utilization (number of still-live bytes) and their age; older and emptier segments are preferred. Once segments are picked for reaping, the block descriptor block (a table of contents of every BU present in the segment) is read from each segment and the blocks in the segment are checked for liveness. Live blocks are copied forward to a new segment, their entries in DI are updated to point to their new location, and once this new location is stable in DI, the old segment is freed.

There is GC for both disk and flash; disk GC does copy-forward, while flash GC never does - flash segments are released in an approximately FIFO order.

index
  • Nimble Datapath module
  • See also BI

Nimble Ref: DsdDataflow

Index implements the data path for generations. For every extant generation, Index maps the addressable blocks to BUs via the BI sub-module. For all stable generations, there is an on-disk BI tree, which is a standard indirect tree (a level-0 (L0) block maps N blocks in the volume, a level-1 block maps N L0 blocks, a level-2 block maps N L1 blocks). For generations which are still open, there is the BI-UB (BI Update Buffer),which tracks changed blocks in the generation. Whenever the Volume Manager decides to close a generation, a new BI tree is generated from the previous generation's BI tree merged with data from the BI-UB. Not surprisingly, this is called a BI merge.

Index also tracks user data for open generations in a subcomponent called DBB (the Dirty Block Buffer). User writes are accumulated into DBB, stored into NVPages and UEs; as NVRAM runs low, this data is flushed from DBB to lower layers. DBB flushes can occur while a generation is still open or once the generation is closed; all user data must be flushed to disk before the BI tree can be generated.

Index does address swizzling of user blocks, as well.

The iSCSI module is the module that communicates with client machines. It exports volumes (and snapshots of volumes) for discovery by clients, receives connection requests, manages multipathing, and handles the details of the iSCSI protocol. Because iSCSI is also aware of connection-level state, it tracks "streams" (ongoing sequences of requests) and identifies when streams are doing sequential accesses. Sequential accesses invoke the logical prefetching mechanism (contained within iSCSI) and mark requests as non-cacheable in flash.

 In the b-series of releases (1.x), ISCSI does discovery directly in DSD; in the 2.x (c-series) timeframe, iSCSI discovery has migrated to the GDD daemon.

iSCSI receives operation requests on a connection for a given Offset and type of operation, and maps that connection to figure out what volume and snapshot (if any) the operation is being done on.

LFS

log-structured file system

  • Nimble Datapath module
  • See also:
    • Indexes
    • CASL

Ref: DsdDataflow

LFS is the log-structured filesystem layer and is used for both hard disks and SSDs in a Nimble array. When a BU is written, LFS assigns the BU an LfsLocation storage location, which can be used later to read the data back. To obtain high write bandwidth, LFS accepts incoming block writes, compresses each block, collects the compressed blocks together into segments - a segment is a contiguous chunk of storage space, which even slow hard disks can write with high bandwidth - and writes a segment at a time. An LfsLocation is a tuple of <segment ID, segment offset>: because of compression, a given data block will not be located on a disk block boundary but somewhere within the segment (LFS has a minimum grain size of 64 bytes.)

Every block written by LFS is self-identifying, with its BU and compressed and uncompressed checksums stored with it. This helps identify misdirected write or corruption problems.

LUNLogical Unit NumberRef: https://en.wikipedia.org/wiki/Logical_unit_number
LUN MappingLUN mapping is the process of associating/assigning a volume to an LU number. This assignment occurs within the context of an initiator group to volume association. The workflow can be broken down as follows:
  • Volume (vol1) is created
  • Initiator group (igroup1) is created
  • For the purposes of data access control, vol1 is associated with igroup1. This association will assign an LU number to vol1 within the context of this association. So all initiators in igroup1 will see vol1 as LUN1 (if 1 is chosen as the LU number).
  • If another initiator group (igroup2) is created and associated with vol1, the system may assign a different LU number even though it is the same volume. However, if the same initiator is part of both igroup1 and igroup2, then the LU number must be the same for a given volume. If this requirement cannot be satisfied (e.g., the LU number is already in use), then the mapping fails, the user is presented an error, and prompted to fix the problem. 
Ref: DataAccessControl

Performed at the storage controller level, LUN masking defines and enforces relationships between LUNs (volumes) and individual severs (initiators). These relationships form the basis of data access control (as enforced by the storage controller) and allow access to a given LUN to be granted to one or more initiators (through the use of initiator groups).Ref: DataAccessControl

There is no single definition for microservices. A consensus view has evolved over time in the industry. Some of the defining characteristics that are frequently cited include:

NIMOS Data Services Internal Concepts

Internal Data Services Concepts

Internal Concept
Definition / Details / Use
Generations
  • view of volume's block space
  • unnamed or internal snapshot
  • can be open (active for reads/writes, metadata in core) or synced (on disk & read-only, metadata on disk)
  • generations have monotonically increasing IDs
  • exactly one open generation for a live volume
  • close a generation when: user takes a snapshot; resource pressure (eg. due to memory consumption associated with state info); time (to avoid data in NVRAM too long)
Branches
  • sequence of generations
  • start from nothing or from a generation on another branch
  • branches to handle interruption or new history (clones, snapshot rollback, replication handover (to another array))
  • generation IDs monotonically increase across branch points but not across sibling branches
VolGen
  • Tree of volume history = column family (VolFam)
  • Branch ID + Generation ID = VolGen
  • VolGens are internal time markers or snapshots
Dedupe Domain
  • store only unique blocks with reference counts
  • set of branches belonging to volumes that dedupe together
  • unrelated branches can share blocks within a dedupe domain
BU/SBNBU = Block Uniqifier (unique Identifier) .. block address
  • Unique address of a block on disk
  • Classic BU: identifies when/where first written
    • VolGen(Branch+Gen) + ldxLBA (metadata) (4+4+8 bytes)
    • LBA = Logical Block Address
  • SBN BU: for dedupe volumes
    • assigned at flush time
    • when block was written to dedupe domain
    • SBN = Sequential Block Number
Block Sharing between generations
  • each generation indexes data as a tree of blocks
  • unchanged data is shared between generations
  • differences between non-snapshot generations go away
Slices & Bins
  • scaling features: deep, up and out
    • scale deep = add more capability to a single array (up to 6 expansion shelves per array)
    • scale up = add more compute to a single array (for performance)
    • scale out = pooling many arrays to use as a single unit for both capacity and performance at a volume level
  • Bin: Unit of rebalancing across arrays (Shard)
    • small map vs fine-grained rebalance
  • Slice: Unit of striping data across bins
    • small to avoid hot spots vs large enough to win on spatial locality (16 MB pre-configured by Nimble)

  • manages flash & memory cache

  • knows where data is located in the scale-out cluster
  • coordinates IO path among arrays in a pool

  • client I/O requests come here
  • stores and reads data off disks

  • assigns IP addresses to NICs, IP failover, monitors disk stats, ...

SDMPD: SNMP Portal

RAD: Role Activation Daemon: shared data availability

  • picks the role of the array in the group
  • runs on active controller only
  • mounts shared partitions on active controller

EMD: Enclosure Management Daemon

  • manages expansion shelf connections to the CMD-managed controller

NVFD: NVRAM Management Daemon

  • manages the non-volatile RAM on the CMD-managed controller
  • save/restore NVRAM content to file on shutdown/startup

Segment: The basic unit of writes onto disks and flash SSDs in a Nimble array. A disk segment is a collection of chunks that spans a complete RAID stripe, so a segment write utilizes all the system disks and gets good write bandwidth out of them. Segment size varies based on the Nimble array model.

Segment:

  • is the smallest unit of disk (flash) writes in a Nimble array
  • spans entire RAID stripe (on-disk) to avoid read-modify-write penalty
    • once a segment is written it will never be modified even if inside blocks die)
  • flash segments have no RAID (Hybrids) per-SSD (for better cache survivability on SSD failure; SSDs are a cache)
  • can be addressed (read) on 64-byte boundaries
  • blocks reside in a segment
  • segment is self describing (header, footer, TOC)


  • garbage collection is required to clean up Swiss cheese content of dead segments.


  • data on disk is broken up into segments
    • a segment is a RAID stripe across all disks
      • on hybrid array HDs (20MB)
      • on AFAa (All Flash Arrays) 10 MB
    • each segment is made up of blocks
      • CASL (Cache Accelerated Sequential Layout) blocksize
        • default is 4 kilobytes (user can pick and almost everyone uses this)
        • up to 32 kilobytes (pick larger for better performance if workload is sequential)

split brain
  • When part of a redundant / highly available system that should operate collaboratively with another part (usually the failover site) loses contact with it's companion, assumes the companion has failed and takes over the operation even though the problem is a connectivity issue and not a failure of the primary service we can have a split brain situation
  • Eg. If the upstream side becomes isolated from the external software and from downstream side (it could be caused by only a single networking failure), but not from hosts, the external software could assume that upstream side went down and trigger switchover. Now both sides take I/O from the hosts but don't replicate it to each other and don't control concurrency across sides. We have a split brain.

Zoning Typically implemented at the hardware level (by using the capabilities of Fibre Channel switches and thus only applicable to the switched fabric topology or FC-SW), zoning allows segregation of a node in general by physical port, name or address. With zoning, a storage switch can be configured to allow a host to talk only with particular ports on a particular storage device (even though more than one storage device may be present in the fabric).
